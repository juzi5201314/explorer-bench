# Explorer Benchmark

对多个 LLM 模型进行代码库探索能力的基准测试。每个模型独立探索 Deno v2.7.10 代码库，使用相同的提示词，由编排代理对输出进行评分。

## 使用方法

### 前置条件

- `omp` CLI 可用
- 网络访问（首次运行需克隆 Deno）

### 快速开始

```bash
# 1. 编写探索提示词
cat > input.md << 'EOF'
# Deno v2.7.10 Codebase Exploration Task
...
EOF

# 2. 确保 Deno 代码库存在（首次自动克隆）
# 3. 向编排代理发送模型列表即可启动
```

模型通过以下命令运行（仅开放探索所需的 5 个工具，`write` / `edit` 被禁用以防止报告写入文件而非 stdout）：

```bash
omp --model "<provider/model_id>" \
    -p "$(cat ./input.md)" \
    --thinking high \
    --tools read,bash,find,lsp,search
```

工具白名单：
 `read`, `find`, `search`, `lsp` — 核心探索：读文件、glob 查找、内容搜索、代码智能
 `bash` — 运行 git/cargo 等系统命令

排除的工具：
 `write`, `edit` — 防止模型将报告写入文件
 `task` — 工具层面强制禁止子代理
 `browser`, `web_search`, `ask`, `notebook`, `python` — 探索任务不需要

### 评分维度

| 维度 | 权重 | 说明 |
|------|------|------|
| 广度 | 25% | 覆盖 6 个必选章节，多子系统分析，文件/目录引用广度 |
| 深度 | 25% | 具体文件路径、函数名、类型名、行号，追溯实际调用链 |
| 质量 | 25% | 结构清晰、组织良好、有洞察力，跨子系统连接 |
| 正确性 | 25% | 抽查 3-5 个声明，与代码库逐条验证 |

总分 = 0.25 × (广度 + 深度 + 质量 + 正确性)，归一化至 1-10。

### 输出结构

```
.
├── input.md              # 探索提示词
├── deno/                 # Deno 代码库
└── outputs/
    ├── <model-id>/
    │   ├── YYYY-MM-DD.md           # 模型输出
    │   └── YYYY-MM-DD.meta.json    # 耗时与退出码
    └── ...
```

### 逐模型点评

**step-3.5-flash (9.75)** — 全面胜出。27KB 输出覆盖全部 6 个必填章节，深入分析了 6 个子系统。精确度极佳：`flags.rs` 声称 15,329 行（实际 15,328）、`http/lib.rs` 声称 1,853 行（实际 1,852）、`permissions/lib.rs` 声称 9,291 行（实际 9,290）——所有关键声明的偏差均在 1 行以内。代码片段、具体行号引用、清晰的架构解释贯穿全文。

**LongCat-2.0-Preview (9.25)** — 紧随其后。23KB 输出结构严谨，涵盖 6 个子系统。分析细致，多次切入 Deno 架构模式。所有验证过的声明均为准确。与第一名的差距主要在于行号精确度稍低。

**qwen3.5-397b-a17b (7.50)** — 扎实的中等水平。16KB 输出覆盖全部章节和 5 个子系统。关键文件路径正确，但缺少代码片段，深度为描述性多于追溯性。

**kimi-k2-thinking (7.25)** — 信息丰富但不够严谨。两处重要事实错误：声称 17 个顶层 crate（实际 ~77 个），以及多次引用不存在的路径 `runtime/permissions/src/lib.rs`。代码结构感好，但具体细节不能尽信。

**qwen3-next-80b-a3b-thinking (6.00)** — 输出有 32% 是系统提示回声，而非实质报告。剩余内容尚可，但存在相同的事实错误。2m14s 是成功模型中最快的。

**gemini-3.1-flash-lite-preview (5.25)** — 仅 32 秒的快速过场，充斥事实错误：引用不存在的 `runtime/main.rs`、`runtime/permissions/mod.rs`、`cli/resolver.rs`。速度快但不可信。

**gemini-3-flash-preview (4.25)** — 太简略。声称将详细报告写入独立文件，但该文件并非输出的一部分。104 秒与内容量不相称。

**deepseek-v4-flash (1.75)** — 14 行摘要，无实质报告。220 秒白费。

**qwen3-next-80b-a3b-instruct / nemotron-3-super-120b-a12b** — 600 秒超时，输出为 0 字节。模型在完整探索 Deno 的负载下无法完成。

**kimi-k2-0905** — API 错误（DEGRADED function），模型端点不可用。

### 分析

**胜出者**: `step-3.5-flash` 在广度、深度和正确性三个相互制约的维度上同时取得最高分，其行数统计表明模型确实精读了关键文件。

**长上下文模型优于速度型**: 前三名均为 400 秒以上的模型。速度最快的模型（32 秒）在所有维度上大幅落后。深度探索不可替代。

**普遍错误**: 权限系统的文件路径（多模型添加了不存在的 `src/` 层级）、crate 数量统计（17 到 77 差异巨大）、行号捏造。

**原始输出**: `./outputs/<model-id>/2026-05-02.md`，每条输出旁均有 `.meta.json` 记录退出码和耗时。
