

### Section 1: High-Level Directory Map
- **.cargo/**: Cargo configuration files (e.g., `config.toml` for build settings)
- **.devcontainer/**: VS Code Dev Container configuration for development environments
- **.github/**: GitHub Actions workflows (CI/CD pipelines, issue templates)
- **cli/**: Core CLI implementation. Contains `main.rs` (entry point), command parsing (`args/`), subcommand handlers (`lsp/`, `run/`, `test/`), and module loading (`module_loader.rs`)
- **ext/**: Extension crates for system capabilities. Each subdirectory is a separate Rust crate (e.g., `ext/fs/` for filesystem ops, `ext/crypto/` for cryptography)
- **libs/**: Shared libraries used across Deno. Includes `core/` (V8 integration), `resolver/` (module resolution), `npm/` (npm compatibility)
- **runtime/**: Runtime implementation. Contains worker management (`worker.rs`), permission system (`permissions/`), and op system (`op.rs`)
- **tests/**: All test suites. Subdirectories include `unit/` (core tests), `integration/` (end-to-end), `node_compat/` (Node.js compatibility)
- **tools/**: Additional CLI tools. Includes `pm/` (package manager), `bundle/` (bundler), `lint/` (linter)
- **Cargo.toml**: Workspace configuration defining all Rust crates and dependencies
- **.gitmodules**: Git submodules (e.g., V8 engine source)
- **rust-toolchain.toml**: Rust toolchain version specification

### Section 2: Crate Dependency Graph
Deno uses a Rust workspace with 50+ crates. Key dependencies:
- **cli/**: Depends on `deno_core`, `deno_graph`, `deno_lint`, `deno_resolver`, and all `ext/*` crates. Uses `deno_core` for V8 integration.
- **ext/crypto/**: Depends on `deno_core` and cryptographic crates (e.g., `aes-gcm`, `chacha20`). Provides `Deno.crypto` API.
- **libs/core/**: Core runtime library. Used by all crates. Implements V8 isolate management, ops registration, and module loading.
- **runtime/**: Depends on `libs/core` and `deno_permissions`. Implements worker lifecycle and permission checks.
- **ext/fs/**: Depends on `deno_core` and `libc`. Provides filesystem ops (`op_fs_read`, `op_fs_write`).
- **libs/resolver/**: Used by `cli/` and `runtime/`. Handles ES module resolution, npm dependencies, and import maps.

Example dependency chain:  
`cli/` → `libs/core` → `ext/fs` → `libc` (for system calls)  
`cli/` → `libs/resolver` → `deno_graph` (for dependency analysis)  

### Section 3: Startup-to-Execution Trace
1. **Entry point**: `cli/main.rs` calls `deno::main()`
2. **Flag parsing**: `cli/lib.rs` parses CLI flags via `args::flags_from_vec_with_initial_cwd`
3. **Runtime initialization**: 
   - `CliFactory` creates `UnconfiguredRuntime` (from `runtime/`)
   - V8 is initialized via `deno_core::v8_init()`
   - Permissions are set up from `runtime/permissions.rs`
4. **Module resolution**: 
   - For `deno run file.ts`: `cli/module_loader.rs` resolves modules using `Resolver` (from `libs/resolver/`)
   - TypeScript is compiled via `tools/tsc.rs` (using `typescript_go_client`)
5. **Execution**: 
   - `worker.rs` creates a `JsRuntime` instance
   - Module source code is loaded into V8 via `deno_core::ops::register_ops()`
   - User script executes in V8 isolate, with ops bridging to Rust via `ext/fs/mod.rs` etc.

Key transition: Control passes from Rust (`cli/lib.rs`) → V8 runtime (`deno_core`) → user JavaScript via `runtime/worker.rs`.

### Section 4: Deep Dive into 3+ Subsystems
**Module Resolution**  
- **Location**: `libs/resolver/`, `cli/module_loader.rs`  
- **Key types**: `Resolver`, `ModuleLoader` trait  
- **Flow**:  
  - `Resolver::resolve()` handles ES modules, npm, and import maps (e.g., `resolver.rs` line 100-200)  
  - Uses `deno_graph` for dependency analysis (e.g., `deno_graph::graph::ModuleGraph`)  
  - npm resolution: `npm_resolver.rs` (via `node_modules` or `npm/` cache)  
- **Integration**: `cli/module_loader.rs` implements `ModuleLoader` for `deno_core` runtime  

**Permission System**  
- **Location**: `runtime/permissions.rs`  
- **Key types**: `Permissions` struct, `PermissionCheck` trait  
- **Flow**:  
  - Each op handler (e.g., `ext/fs/mod.rs:op_fs_read()`) checks permissions via `permissions.check_read()`  
  - Permission checks are defined in `runtime/permissions.rs` (e.g., `check_read()` line 150-180)  
  - Permissions are granted via CLI flags (`--allow-read`, `--allow-net`) and stored in `Permissions` struct  
- **Integration**: `runtime/worker.rs` passes `Permissions` to `JsRuntime` during creation  

**Runtime Ops System**  
- **Location**: `libs/ops/`, `ext/*/mod.rs`  
- **Key types**: `OpDecl`, `OpState`  
- **Flow**:  
  - Ops are registered via `deno_core::ops::register_ops()` in `ext/*/mod.rs` (e.g., `ext/fs/mod.rs:op_fs_read` at line 300-350)  
  - JavaScript calls ops via `Deno.core.ops` (e.g., `Deno.readSync()` → `op_fs_read` in Rust)  
  - Data marshaling between V8 and Rust handled via `serde_v8` (in `libs/serde_v8/`)  
- **Integration**: `runtime/worker.rs` initializes op handlers during `JsRuntime` setup  

### Section 5: Design Decisions & Architectural Patterns
**1. Workspace-based Rust crates**  
- **Decision**: Split Deno into multiple Rust crates (e.g., `cli/`, `ext/fs/`, `libs/core/`)  
- **Rationale**: Enables independent development, reduces compile times, and isolates security-critical components (e.g., filesystem ops in `ext/fs/` are sandboxed)  
- **Implementation**: Defined in `Cargo.toml` workspace members. Each crate has its own `Cargo.toml` (e.g., `ext/fs/Cargo.toml` lists dependencies)  

**2. Permission checks at op level**  
- **Decision**: Validate permissions in Rust op handlers, not in JavaScript  
- **Rationale**: Prevents JavaScript from bypassing security checks. Ensures all system calls are validated before execution  
- **Implementation**: In `ext/fs/mod.rs`, `op_fs_read` first calls `permissions.check_read()` before reading files  

**3. V8 isolate management via deno_core**  
- **Decision**: Centralize V8 interaction in `libs/core/`  
- **Rationale**: Abstracts V8 complexity, provides consistent API for all crates. Avoids duplicate V8 initialization code  
- **Implementation**: `deno_core::JsRuntime` in `libs/core/` handles isolate creation, module loading, and op registration  

**4. TypeScript compilation via Go-based compiler**  
- **Decision**: Use Go compiler (`typescript_go_client`) instead of TypeScript compiler in Rust  
- **Rationale**: Leverages mature TypeScript compiler, avoids maintaining a Rust-based TS transpiler  
- **Implementation**: `tools/tsc.rs` spawns Go process to compile TS files, then feeds output to V8  

### Section 6: Critical Files Index
1. **cli/main.rs**: Entry point for Deno CLI (calls `deno::main()`)  
2. **cli/lib.rs**: Main CLI logic (handles subcommands, flag parsing, runtime initialization)  
3. **runtime/worker.rs**: Manages worker threads and V8 isolate lifecycle  
4. **ext/fs/mod.rs**: Filesystem operations implementation (e.g., `op_fs_read`, `op_fs_write`)  
5. **libs/core/mod.rs**: Core runtime library (V8 integration, ops registration, module loading)  
6. **runtime/permissions.rs**: Permission checking logic (e.g., `check_read()`, `check_net()`)  
7. **cli/module_loader.rs**: Module resolution and loading logic (ES modules, npm, import maps)  
8. **libs/resolver/mod.rs**: Module resolution implementation (handles `import_map.json`, npm dependencies)  
9. **runtime/op.rs**: Base implementation for op handlers (marshaling between JS and Rust)  
10. **tools/tsc.rs**: TypeScript compilation interface (spawns Go compiler process)  
11. **ext/crypto/mod.rs**: Cryptographic operations (e.g., `Deno.crypto.subtle` implementation)  
12. **ext/net/mod.rs**: Network operations (TCP/UDP sockets, HTTP server)  
13. **libs/serde_v8/mod.rs**: Data serialization between V8 and Rust  
14. **tests/unit/mod.rs**: Core unit tests (e.g., V8 integration, ops system)  
15. **Cargo.toml**: Workspace configuration defining all crates and dependencies  
16. **runtime/permissions.rs**: Permission system definition (critical for security)  
17. **cli/args/mod.rs**: CLI argument parsing logic (e.g., `--allow-read`, `--unstable`)  
18. **libs/core/inspector.rs**: Debugger protocol implementation (LSP integration)  
19. **ext/web/mod.rs**: Web API implementation (e.g., `fetch`, `WebSockets`)  
20. **tools/compile.rs**: Standalone binary compilation logic (creates executable from JS/TS)
