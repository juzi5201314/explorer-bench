## 5. Design Decisions & Architectural Patterns

### Decision 1: Extension-Based Plugin Architecture

**What**: Instead of monolithic runtime, Deno is built from 25+ extension crates. Each extension is a self-contained module that registers ops and JS sources via `deno_core::extension!` macro.

**Why**: 
- Clear separation of concerns (fs, http, crypto are independent)
- Enables feature flags and lazy initialization
- Allows building custom runtimes by selecting extensions
- Reduces compile times for debugging individual parts

**Where**: `libs/core/extensions.rs`, `runtime/worker.rs:common_extensions()`, every `ext/*/lib.rs`

---

### Decision 2: Ops as the JS↔Rust Boundary

**What**: All JavaScript-to-Rust calls go through "ops" defined with `#[op2]` macro. No direct FFI or raw V8 calls from higher-level code.

**Why**:
- Type-safe (macro generates conversion code)
- Fast (uses V8's Fast API when possible)
- Supports async (ops can return `Future`)
- Centralized error handling and permission checks
- Easy to instrument (metrics, stack traces)

**Where**: `libs/ops/op2/` (macro definition), `ext/*/ops.rs` (op implementations), `libs/core/ops.rs` (op dispatch)

---

### Decision 3: Snapshot-Based Startup

**What**: Deno can be compiled with a pre-snapshotted JavaScript runtime (`deno_snapshots` crate). The snapshot contains pre-compiled V8 heap with all built-in JS sources.

**Why**:
- Fast startup (no JS parsing/compilation)
- Reduces binary size (compressed snapshot)
- Allows tree-shaking of built-in JS

**Where**: `cli/snapshot/` (snapshot creation), `cli/build.rs` (embedding), `libs/core/runtime.rs` (loading)

---

### Decision 4: Graph-Based Module Resolution

**What**: Deno builds a full module dependency graph (`deno_graph` crate) before executing, rather than loading on demand.

**Why**:
- Enables type checking across entire program
- Detects circular dependencies early
- Supports import maps and npm interop correctly
- Allows bundling and compilation

**Where**: `libs/resolver/graph/`, `cli/graph_container.rs`, `cli/graph_util.rs`

---

### Decision 5: Permission Prompts at First Use

**What**: Permissions default to "Prompt". On first access, user is asked interactively (or via flag). After prompt, permission may be granted for entire process.

**Why**:
- Secure by default (matches Deno's security model)
- Good UX (prompts are specific to operation)
- Supports automation via flags (--allow-read, etc.)

**Where**: `runtime/permissions/lib.rs:PermissionState::check()`, `runtime/permissions/prompter.rs`
