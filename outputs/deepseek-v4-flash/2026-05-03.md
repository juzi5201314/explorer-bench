**依赖方向：** `deno_core` 在底层，是所有代码的基础。`runtime/` crate 引入 `deno_core` 和所有扩展。`cli/` crate 包装所有内容。`deno_lib`（`cli/lib/`）是 `cli/` 和 `runtime/` 之间的过度层——它定义了 `LibMainWorker` 工厂，供 `cli/` 使用，同时也被 `deno compile` 生成的可执行文件使用。

### 关键架构层（从底层到顶层）：

1. **V8 层** — `rusty_v8` crate（外部）、`deno_ops` proc-macro、`serde_v8` 序列化
2. **核心层** — `deno_core`（`libs/core/`）：`JsRuntime`、`OpState`、模块加载、事件循环、检查器
3. **扩展层** — `ext/*/`：通过 `deno_core::extension!` 注册的 30+ crate，每个提供 op + JS 源文件
4. **运行时层** — `runtime/`：`MainWorker`、权限系统、运行时 op（worker 创建、bootstrap）
5. **库层** — `cli/lib/`（`deno_lib`）：`LibMainWorkerFactory`、共享 args/npm/resolver 逻辑
6. **CLI 层** — `cli/`：子命令分派、CLI 工厂、LSP、TypeScript 编译

## 第4节：3+ 子系统深度剖析

### 4.1 模块解析与加载子系统

**所在位置：** 分布在多个 crate 中：
- `libs/resolver/` — 核心解析器：工作区解析、import maps、npm 集成
- `cli/module_loader.rs` — 将解析器连接到 V8 `ModuleLoader` trait
- `libs/core/modules/` — V8 模块接口：`ModuleLoader`、`ModuleMap`、`recursive_load`
- `libs/node_resolver/` — Node 算法实现
- `libs/npm/` — npm 版本/注册表解析

**关键类型：**
- `deno_core::ModuleLoader` (`libs/core/modules/loaders.rs`) — 由 V8 调用以解析/加载模块的 trait
- `CliModuleLoaderFactory` (`cli/module_loader.rs`) — 创建用于 main 和 worker 的 CLI 模块加载器
- `WorkspaceResolver` (`libs/resolver/workspace.rs`, 3290 行) — 核心解析引擎：处理 import maps、deno.json、node_modules、npm 重映射、JSR
- `ModuleGraphBuilder` (`cli/graph_util.rs`) — 使用 `deno_graph` 构建整个依赖图
- `CliFileFetcher` (`cli/file_fetcher.rs`) — 获取具有权限检查的远程/本地模块
- `ParsedSourceCache` (`libs/resolver/cache/`) — 缓存已解析的源文件和 AST

**工作流：**
1. V8 遇到 `import` 时调用 `ModuleLoader::resolve()` + `ModuleLoader::load()`
2. CLI 的模块加载器查询 `WorkspaceResolver`，后者检查 import maps → deno.json → npm → node_modules
3. 解析后的说明符传递给 `ModuleGraphBuilder`，后者递归地遍历所有依赖
4. 每个模块进行媒体类型检测（TS/JS/JSX/JSON）、缓存、可选的类型检查
5. `ModuleLoader` 将转译后的源码返回给 V8 进行编译

**连接：** 权限系统（检查每个获取的说明符）、TypeScript 编译器（类型检查）、npm 安装器（解析 npm 依赖）、deno.json 配置（路径/导入映射）。

### 4.2 权限系统

**所在位置：**
- `runtime/permissions/lib.rs` — 核心类型和实现（9291 行）
- `runtime/permissions/prompter.rs` — 用户提示 UI
- `runtime/permissions/broker.rs` — 外部权限代理
- `runtime/permissions/runtime_descriptor_parser.rs` — 描述符解析
- `runtime/permissions/which.rs` — 系统二进制解析
- `runtime/ops/permissions.rs` — 权限 op（授权/撤销/查询）

**关键类型：**
- `PermissionsContainer` (`runtime/permissions/lib.rs`) — 线程安全的权限封装；`Arc<parking_lot::Mutex<Permissions>>`
- `Permissions` — 实际权限状态：`read`、`write`、`net`、`env`、`run`、`sys`、`ffi` 等，每个包含允许列表/拒绝列表
- `PermissionDeniedError` — 实现 `#[class("NotCapable")]` 的 `JsError`，因此 JS 方面会看到 `NotCapable` 错误类型
- `PromptResponse` — `Allow` / `Deny` / `DenyOnce`
- `OtelAuditFn` / `AuditSink` — 通过 OpenTelemetry 或文件的可选审核日志记录

**架构：**
- 权限检查发生在每个 op 内部（例如 `op_open` 调用 `permissions.check_read()`）
- `PermissionsContainer` 方法在 `Permissions` 内部获取锁
- 在非提示模式下的不匹配会立即返回 `PermissionDeniedError`
- 在提示模式下会显示一个交互式提示（颜色、堆栈跟踪）
- 外部权限代理允许父进程决定权限
- 审核日志记录支持 OpenTelemetry 跟踪和文件输出

**连接：** 每个 I/O（文件、网络、env）和 FFI op 都通过 `PermissionsContainer` 调用权限检查。

### 4.3 运行时操作码系统（JS 到 Rust 桥接）

**所在位置：**
- `libs/ops/` — `#[op2]` 过程宏（DSL 编译器）
- `libs/core/ops.rs` — `OpCtx`、`OpState`、`OpDecl` 运行时
- `libs/core/ops_builtin.rs` — 核心内置 op（close、read、write、resources、wasm 流式传输）
- `libs/core/ops_builtin_v8.rs` — V8 内置 op（timer、microtask、编码、eval、promise hooks、leak 跟踪）
- `libs/core/ops_builtin_types.rs` — JS 类型检查 op（is_array_buffer、is_date…）
- `libs/core/ops_metrics.rs` — 每个 op 的指标框架
- `libs/core/runtime/op_driver/` — 操作码分派执行器

**关键类型：**
- `#[op2]` proc-macro (`libs/ops/op2/`) — 定义 op 的 DSL；支持 `#[op2(fast)]` 用于 V8 快速 API（零开销调用）、`#[string]`、`#[serde]`、`#[global]`、`#[state]` 参数属性
- `OpDecl` — 描述 op 的静态声明（名称、参数、是否为快速 API）
- `OpCtx` (`libs/core/ops.rs`) — 每个 realm 的上下文：op ID、未加工的隔离区指针、状态、指标函数
- `OpState` — 每个隔离区的可变状态（通过 `Rc<RefCell<OpState>>` 访问）。作为类型映射的 `GothamState` 的集合，包含资源表、op ID 注册表
- `deno_core::extension!` 宏 (`libs/core/extensions.rs`) — 将一组 op + JS 源文件声明为可组合单元

**架构：**
- op 通过 `#[op2]` 属性宏定义，该宏展开为一个 Rust 函数 + 一个 `OpDecl`
- 函数要么是同步的（直接结果），要么是异步的（返回 `Future`）
- `#[op2(fast)]` 使用 V8 快速 API（`v8::fast_api::CFunction`），实现零序列化的 V8 ≤ → Rust 调用——不需要 `serde_v8` 转换
- 所有 op 都在 V8 Isolate 创建时注册，通过 `Extension` 组合
- 调用时，JSON 参数通过 `serde_v8` 反序列化（或直接用于快速 API 的原语），函数执行，结果序列化回 V8
- 每个 op 可以使用 `#[op2(stack_trace)]` 启用堆栈跟踪收集
- 支持可重入检测（调试断言）

**连接：** 每个扩展（web、fetch、http、ffi 等）使用 `deno_core::extension!` 注册其 op。运行时 op（`runtime/ops/`）实现 bootstrap、权限、worker 主机和控制。

### 4.4 HTTP 服务器实现

**所在位置：**
- `ext/http/lib.rs` (1853 行) — 核心 HTTP 服务器实现
- `ext/http/service.rs` — Hyper 服务适配器
- `ext/http/http_next.rs` — 下一个开发版本 HTTP 处理
- `ext/http/request_body.rs` / `response_body.rs` — 请求/响应主体的资源包装器
- `ext/http/00_serve.ts` / `01_http.js` — JS API 层

**关键类型：**
- `deno_http::HttpClientResource` — 由 hyper 支持的 HTTP 客户端
- `deno_http::HttpServer` — 由 hyper 和 tokio 支持的 HTTP 服务器
- `HyperService` (`service.rs`) — 实现 `hyper::service::Service`
- `NetworkStream` (`ext/net/`) — TCP/TLS 抽象，包装在 hyper 的 `TokioIo` 中
- `RequestBodyResource` — 将主体流暴露为 Deno 资源
- `ResponseBodyResource` — 将主体写入流式传输

**架构：**
- 暴露于 JS 的 `Deno.serve()` 绑定到 Rust op `op_http_serve` 和 `op_http_fetch`
- 服务器使用 hyper 1.x（`hyper::server::conn::http1` / `http2`）
- 连接被接受为 TCP 流，包装在 `TokioIo<NetworkBufferedStream>` 中，然后传递给 hyper
- 请求作为资源暴露给 JS；JS 创建 Response 对象，流回 Rust
- 支持 HTTP/1.1 和 HTTP/2
- 压缩使用自定义 `fly_accept_encoding` 协商（Brotli、Gzip、Deflate）
- WebSocket 升级通过 `ext/websocket` 处理，使用 `fastwebsockets` crate

**连接：** 使用 `deno_net` 进行 TCP/TLS，使用 `ext/websocket` 进行升级，使用 `deno_tls` 进行 TLS 配置，使用 `deno_telemetry` 进行 HTTP 指标。

### 4.5 npm/Node.js 兼容层

**所在位置：**
- `ext/node/` — node: 内置模块实现（Rust 中的 op + polyfill JS）
- `ext/node/lib.rs` — crate 根目录
- `ext/node/build.rs` — 生成 Node.js 模块列表
- `libs/node_resolver/` — 遵循 Node 算法的模块解析
- `libs/npm/` — npm 注册表和版本解析
- `libs/npm_cache/` — npm 包缓存
- `libs/npm_installer/` — 包安装和生命周期脚本
- `cli/npm.rs` — CLI 级别的 npm 集成：`CliNpmCache`、`CliNpmResolver`、生命周期脚本

**关键类型：**
- `NodeRequireLoader` / `NodeRequireLoaderRc` — `require()` 实现
- `NpmResolver` (`libs/resolver/npm.rs`) — npm 包解析 trait
- `ManagedNpmResolver` — 解析/安装/缓存 npm 包的完整实现
- `NpmInstaller` (`libs/npm_installer/`) — 包下载、提取、生命周期脚本、bin 链接
- `CliNodeResolver` (`cli/node.rs`) — CLI 端的 node 入口点解析
- `ExtNodeSys` (`ext/node/`) — Node 兼容性系统描述符

**架构：**
- Node.js 兼容性在 Rust 层内部实现（op + C 函数调用），而不是通过 JS shim
- `require()` 被实现为 Rust op，使用 Node 解析算法（通过 `node_resolver`）
- npm 包解析支持两种模式：管理（DENO_DIR 中的全局缓存）和 BYONM（自带 node_modules）
- 生命周期脚本在沙箱化子进程中执行，具有超时和权限控制
- `--node-modules-dir` 标志控制 node_modules 目录行为
- CJS → ESM 互操作通过 `IsCjsResolutionMode` 和 CJS 轨迹跟踪器处理
- Node.js `node:` 内置模块实现分布在 `ext/node/` 中的多个文件中

**连接：** 模块加载器（CJS/ESM 混合）、解析器工作区、权限系统（脚本执行）、文件系统层。

## 第5节：设计决策与架构模式

### 5.1 基于扩展的组合架构

**决策：** 所有功能域（web、fetch、http、ffi、crypto…）都被实现为可组合的 `Extension`，通过 `deno_core::extension!` 宏注册。

**原理：** 这允许清晰的关注点分离、可选的编译时包含，以及通过仅添加所需扩展来创建自定义运行时（例如 `deno compile` 可执行文件）的能力。`RuntimeOptions` 通过插入扩展向量来控制包含哪些功能。

**证据：** 每个 `ext/*/` crate 都包含一个 `deno_core::extension!` 调用（例如 `ext/ffi/lib.rs:68`、`ext/http/lib.rs` 最后的调用）。运行时 `common_extensions()` 函数 (`runtime/worker.rs:450+`) 将各个扩展组合到最终列表中。

### 5.2 基于 `#[op2]` proc-macro 的运行时代码生成

**决策：** 使用自定义 proc-macro (`#[op2]`) 和 `serde_v8` 来定义 V8 ↔ Rust 绑定，而不是手动编写 V8 API 的胶水代码。

**原理：** 这消除了模板代码，通过静态 `OpDecl` 实现编译时优化，并支持 `#[op2(fast)]`，允许使用 V8 快速 API（Fast C API）实现零开销调用——无需序列化，无需中间缓冲区。

**证据：** `libs/ops/op2/` 是 proc-macro 的实现。`#[op2(fast)]` op 的展开生成 `v8::fast_api::CFunction` 注册（见 `ops_builtin_v8.rs` 中如 `op_timer_schedule` 和 `op_add` 的示例）。`serde_v8` crate 处理通过 `#[serde]` 属性标记的参数的序列化和反序列化。

### 5.3 分层多阶段模块解析

**决策：** 模块解析发生在多个层，信息通过工作区分步骤积累：
1. `CliFactory` 的惰性初始化
2. 工作区发现（`WorkspaceResolver`）
3. import map 加载
4. Node 模块解析
5. npm 注册表解析和安装
6. 完整的模块图构建（通过 `deno_graph`）

**原理：** 每个步骤都依赖于前一步的数据。分裂允许：
- 惰性初始化：仅在需要时获取配置、安装 npm 包
- 明确的错误边界：每个阶段可以产生特定诊断
- 跨程序缓存：npm 快照、导入映射、工作区配置

**证据：** `CliFactory` (`cli/factory.rs`) 使用 `Deferred<T>`（OnceCell）模式实现惰性初始化。调用 `factory.workspace_resolver().await?` 在首次调用时触发异步解析，否则返回缓存的结果。`deno_resolver` 将工作区解析、npm 解析和导入映射分离到独立模块。

### 5.4 `MaybeArc` 同步抽象

**决策：** `libs/maybe_sync/` crate 提供 `MaybeArc<T>` 类型，该类型在单线程上下文中充当 `Rc<T>`，在多线程上下文中充当 `Arc<T>`，由编译时特征选择。

**原理：** Deno 在 tokio 运行时上运行，该运行时使用多线程执行器，但有些数据结构不需要跨越线程边界。`MaybeArc` 允许在不进入 Arc 的开销的情况下使用相同的数据结构，同时为多线程使用提供编译时安全性。这减少了从多线程获得的同步开销（对具有高争用的数据结构有显著性能影响）。

**证据：** `libs/maybe_sync/` crate 包含 `MaybeArc`、`MaybeBox`、`MaybeDashMap` 和 `MaybeRefCell`。在整个代码库中使用：例如 `libs/resolver/lib.rs:50` 中的 `WorkspaceResolverRc` 定义为 `MaybeArc<WorkspaceResolver<TSys>>`。

### 5.5 快照优先的启动策略

**决策：** V8 运行时是提前进行快照的，包含所有内置 JS 模块和核心运行时状态。快照在构建时生成并嵌入二进制文件。运行时 JS 源文件通过 `include_js_files_for_snapshotting` Cargo 特性有条件地包含。

**原理：** 快照大大减少了启动时间——它们避免了在每次启动时解析和编译核心运行时 JS 文件。V8 快照包含包括解析后的 AST 和已编译代码在内的堆状态，这意味着运行时原型是即时可用的。`deno compile` 可执行文件使用此机制创建独立的二进制文件。

**证据：**
- `cli/build.rs` 构建时压缩 Declarations 并生成快照
- `cli/snapshot/` crate 管理快照创建/加载
- `runtime/snapshot.rs` 提供加载和保存快照的接口
- `RuntimeOptions::startup_snapshot` 控制快照加载
- `deno_snapshots` crate（版本 `0.58.0`）专门处理快照管理
- `#[cfg(feature = "only_snapshotted_js_sources")]` 在快照构建后强制执行无 JS 源文件
