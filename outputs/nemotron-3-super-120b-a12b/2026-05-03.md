# Deno v2.7.10 Codebase Exploration Report

## Section 1: High-Level Directory Map

The Deno repository follows a structured layout with clear separation of concerns:

### Root Directory Contents:
- **cli/** - Contains the main Deno executable and command-line interface
- **ext/** - Built-in extensions providing standard library functionality (fs, net, http, etc.)
- **libs/** - Shared libraries used across the codebase (core, config, npm, etc.)
- **runtime/** - Core runtime components (permissions, features, etc.)
- **tests/** - Test suite organized by type (unit, integration, bench, etc.)
- **tools/** - Development and maintenance tools

### Key Directory Purposes:

**cli/**: Entry point for the deno executable. Contains:
- `main.rs` - Minimal entry point that delegates to `cli/lib.rs::main()`
- `lib.rs` - Main CLI logic with subcommand handling (run, test, bundle, etc.)
- `args/` - Command-line argument parsing
- `worker.js` - Web Worker implementation
- `ops/` - Custom operations exposed to JavaScript

**ext/**: Built-in extensions implementing web platform APIs and Deno-specific functionality:
- Each subdirectory (fs, http, net, etc.) implements a specific Web API or Deno namespace
- Follows a pattern of `lib.rs` defining the extension and implementation files

**libs/**: Shared utility libraries:
- `core/` - Deno core abstraction layer over V8 and Tokio
- `config/` - Configuration loading and parsing
- `npm/` - Node.js compatibility utilities
- `ops/` - Low-level operations shared between CLI and runtime

**runtime/**: Core runtime components:
- `permissions/` - Permission system implementation
- `features/` - Feature flags management
- `js.rs` - JavaScript runtime setup

## Section 2: Crate Dependency Graph

Based on the workspace `Cargo.toml`, Deno is organized as a Rust workspace with approximately 60+ crates. Here are the main crates and their responsibilities:

### Core Crates:
1. **deno_cli** (cli crate) - Main executable, argument parsing, subcommand dispatch
2. **deno_core** (libs/core) - Fundamental abstractions over V8, event loop, op system
3. **deno_runtime** (runtime) - JavaScript runtime with Deno namespace APIs
4. **deno_lint** - Linting functionality
5. **deno_ast** - TypeScript/JavaScript parsing and transpilation
6. **deno_graph** - Module dependency graph construction
7. **deno_npm** - npm package resolution and installation

### Extension Crates (ext/):
Each implements specific Web APIs or Deno namespaces:
- **deno_fs** - File system API
- **deno_net** - Networking APIs (TCP, UDP, TLS)
- **deno_http** - HTTP client and server
- **deno_web** - Web APIs (DOM, events, etc.)
- **deno_websocket** - WebSocket implementation
- **deno_kv** - Key-value storage
- **deno_crypto** - Cryptographic operations

### Library Crates (libs/):
- **deno_config** - Configuration parsing (deno.json, import maps)
- **deno npm_crates** - npm resolution, installation, caching
- **deno_lockfile** - Lockfile management
- **deno_resolver** - Module resolution (including npm and JSR)
- **deno_semver** - Semantic versioning
- **deno_inspector_server** - DevTools/debugger integration

### Dependency Relationships:
- **deno_cli** depends on: deno_core, deno_runtime, deno_args, deno_tokens, deno_config
- **deno_runtime** depends on: deno_core, deno_permissions, deno_tokens
- **deno_core** depends on: v8, tokio, rusqlite (for snapshots), hyper (for HTTP)
- **deno_ext_*** crates depend on: deno_core, deno_runtime (for ops)
- **libs/npm_*** crates depend on: deno_core, deno_npm, deno_lockfile

The workspace uses a `[workspace.dependencies]` section to pin versions and provide path dependencies for local crates.

## Section 3: Startup-to-Execution Trace

Based on reading `cli/lib.rs::main()` function:

### Entry Point:
1. **`deno/cli/main.rs`** - Simply calls `deno::main()`
2. **`deno/cli/lib.rs::main()`** - Actual entry point with initialization sequence

### Initialization Steps:
1. **Setup profiling** (if dhat-heap feature enabled)
2. **Set panic hook** via `setup_panic_hook()`
3. **Initialize logging** via `init_logging(None, None)`
4. **Raise file descriptor limit** via `util::unix::raise_fd_limit()`
5. **Windows-specific setup**: Ensure stdio open, disable stdio inheritance, enable ANSI colors
6. **Set permission prompt callbacks** using DrawThread for hiding/showing during prompts
7. **Maybe setup permission broker** via `maybe_setup_permission_broker()`
8. **Install Rustls crypto provider** (AWS-LC)
9. **Parse command-line arguments**: Collect args via `env::args_os()`
10. **Handle wait-for-debugger functionality** on Unix via socket communication
11. **Determine initial working directory**
12. **Initialize V8 platform** on parent thread (required for V8 11.6+ PKU feature)
13. **Resolve flags and initialize** via `resolve_flags_and_init()`:
    - Handle shell completions if COMPLETE env var set
    - Parse CLI arguments into Flags struct
    - Handle version display and help
    - Snapshot environment variables if watching
    - Process env-file loading
14. **Initialize V8** if not already done by waiter
15. **Run subcommand** via `run_subcommand()` with flags, runtime, and worker roots
16. **Exit** with proper exit code or error handling

### Control Flow to User Code:
For `deno run` subcommand:
1. **`tools::run::run_script()`** called with WorkerExecutionMode::Run
2. **Creates MainWorker** via CliFactory
3. **Bootstraps the worker** which:
    - Creates JsRuntime (V8 instance) with extensions
    - Registers Deno namespace objects (Deno.*)
    - Sets up module loader, permissions, etc.
4. **Loads and executes the main module**:
    - Uses module loader to fetch, compile, and instantiate the entry point
    - Transpiles TypeScript if needed via deno_ast
    - Applies import maps if configured
    - Executes the module's top-level code
5. **Returns to event loop** which runs until no more async tasks

The transition from Rust to JavaScript occurs when V8 executes the compiled module code, with Rust providing the runtime environment and APIs via ops (operations).

## Section 4: Deep Dive into 3+ Subsystems

### Subsystem 1: Module Resolution and Loading

**Location**: 
- `libs/resolver/` - Core resolver implementation
- `cli/resolver.rs` - CLI-specific resolver wrapper
- `cli/module_loader.rs` - Module loading pipeline
- `libs/node_resolver/` - Node.js-compatible resolution

**Key Components**:
- `GraphEdge` - Represents a dependency relationship between modules
- `ModuleSpecifier` - Normalized URL representation of modules
- `ModuleLoader` trait - Abstracts loading strategies (fs, remote, data, etc.)
- `CliModuleLoader` - CLI implementation combining multiple loaders
- `NpmResolver` - Resolves bare specifiers to node_modules packages
- `SloppyImportsHandler` - Handles non-standard import behaviors

**Connections**:
- Used by CLI during `deno info`, `deno cache`, `deno run`
- Integrates with deno_graph for dependency graph construction
- Works with npm resolution for `node:` specifiers and package imports
- Feeds into transpilation pipeline for TypeScript files

### Subsystem 2: Permission System

**Location**:
- `runtime/permissions/` - Core permission implementation
- `cli/lib.rs` - Permission broker setup and prompt callbacks

**Key Components**:
- `Permission` enum - Represents different permission types (read, write, net, env, etc.)
- `PermissionState` - Granted, denied, or prompt states
- `Permissions` container - Manages collection of permissions
- `PermissionCallback` - Trait for prompting users
- `Permitter` - Checks if operations are allowed

**Key Functions**:
- `check_permission()` - Verifies if an operation is allowed
- `revoke_permission()` - Removes a previously granted permission
- `prompt_for_permission()` - Asks user via DrawThread UI

**Connections**:
- Used by all ops (operations) that interact with system resources
- Integrated with worker creation to establish initial permissions
- Allows fine-grained control via `--allow-*` flags
- Permission broker enables secure permission handling in subprocesses

### Subsystem 3: HTTP Server Implementation

**Location**:
- `ext/http/` - HTTP client and server implementations
- `ext/net/` - TCP/TLS foundations
- `cli/tools/serve.rs` - `deno serve` subcommand

**Key Components**:
- `HttpServer` - High-level server API
- `HttpConn` - Connection-level handling
- `ServeHandler` - Trait for handling HTTP requests
- `ListenOptions` - Configuration for binding addresses
- `HttpClient` - Fetch API implementation

**Key Functions**:
- `Deno.serve()` - Main API for creating HTTP servers
- `listen_tcp()`/`listen_tls()` - Low-level socket creation
- `HttpConn::serve()` - Connection handling loop

**Connections**:
- Built on top of TCP/TLS from `ext/net/`
- Uses Tokio for async I/O
- Integrates with permission system (requires `--allow-net`)
- Provides both client (fetch) and server (Deno.serve) APIs
- Supports HTTP/1.1 and HTTP/2 via hyper and h2 crates

### Subsystem 4: Runtime Ops System (JS-to-Rust Bridge)

**Location**:
- `libs/ops/` - Operation definitions and implementations
- `runtime/js.rs` - JS runtime setup registering ops
- `ext/*/ops.rs` - Extension-specific operations

**Key Components**:
- `OpDecl` - Declaration of an operation (name, signature, etc.)
- `OpState` - Mutable state accessible to ops
- `Op` trait - Async operation execution
- `JsonOp`/`SyncOp` - Specialized op types for JSON sync/async operations

**Key Functions**:
- `op_async()` - Creates async operation
- `op_sync()` - Creates synchronous operation
- `op2()` - Version 2 op system with better ergonomics

**Connections**:
- Bridge between JavaScript (V8) and Rust
- Each Deno API (Deno.readFile, Deno.net.connect, etc.) is implemented as an op
- Used by extensions to expose functionality to JavaScript
- Permission checks typically happen within ops implementations

## Section 5: Design Decisions & Architectural Patterns

### 1. **Secure-by-Default Permission Model**
- **Decision**: All system access requires explicit permission grants
- **Rationale**: Prevents malicious scripts from accessing filesystem/network without consent
- **Implementation**: Permission checks in every op that accesses resources; CLI flags (`--allow-read`, etc.) grant permissions at startup

### 2. **Web Standard APIs First**
- **Decision**: Implement web platform APIs (fetch, WebSocket, etc.) before Deno-specific ones
- **Rationale**: Familiarity for web developers, reduces learning curve
- **Implementation**: `ext/web/` implements standard web APIs; Deno namespace for Deno-specific features

### 3. **Single Executable with Built-in Standard Library**
- **Decision**: No separate stdlib distribution; standard library compiled into binary
- **Rationale**: Simplifies deployment and versioning; no dependency hell
- **Implementation**: `ext/` directories compiled into deno_binary; `std` import paths map to built-in extensions

### 4. **Op-Based Extension System**
- **Decision**: Use typed async/sync operations as bridge between JS and Rust
- **Rationale**: Type safety, clear boundaries, easy to audit and test
**Implementation**: `OpDecl` definitions in each extension; registered with V8 via `Extension` struct

### 5. **Modular Workspace with Clear Boundaries**
- **Decision**: Separate crates for CLI, runtime, core, extensions, and libraries
- **Rationale**: Maintainability, testability, clear dependency management
- **Implementation**: `Cargo.toml` workspace with path dependencies; each crate has single responsibility
