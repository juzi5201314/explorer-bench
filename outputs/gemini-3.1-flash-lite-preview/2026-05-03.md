## Section 1: High-Level Directory Map

The Deno repository is structured as a Rust workspace. Key directories include:

*   **`cli/`**: Contains the Deno CLI implementation, including entry point (`main.rs`), subcommands (`tools/`), argument parsing (`args/`), and high-level module loading/resolution logic.
*   **`ext/`**: Houses Deno's extensions. These are mostly self-contained crates that implement runtime APIs using Deno's op-system (e.g., `ext/fetch`, `ext/fs`, `ext/http`, `ext/node` for Node.js compatibility).
*   **`runtime/`**: Core runtime environment logic, including worker management (`worker.rs`), permission checks (`permissions.rs`), and bootstrap logic.
*   **`libs/`**: A collection of reusable libraries underpinning the runtime and CLI, such as `core/` (Deno's V8-to-Rust bridge), `npm/` (npm compatibility), and `eszip/` (module bundling).
*   **`tests/`**: Integration tests and benchmarking utilities.

## Section 2: Crate Dependency Graph

Deno utilizes a workspace-based dependency management approach, with central dependency definitions in the root `Cargo.toml`.

*   **`deno_core` (`libs/core`)**: The absolute foundation. Provides the V8 isolate, snapshotting, and the basic JS-to-Rust "op" (operation) mechanism.
*   **`deno_runtime` (`runtime/`)**: Depends on `deno_core` and various extensions in `ext/`. It provides the high-level `Worker` abstraction used for executing scripts.
*   **`cli/`**: Depends on `deno_runtime`, `ext/*`, `libs/*`, and specialized modules like `deno_ast` and `deno_graph`. It coordinates user commands and bootstraps the runtime.
*   **Extensions (`ext/*`)**: Primarily depend on `deno_core` for implementing ops, and sometimes on other extensions or libraries to fulfill specific API requirements (e.g., `ext/http` depends on `hyper` and `ext/net`).

## Section 3: Startup-to-Execution Trace

1.  **Entry Point (`cli/main.rs`)**: `main()` is the entry point. It parses CLI arguments using `cli/args/` and delegates to the appropriate subcommand (e.g., `run`, `test`, `eval`).
2.  **CLI Initialization**:
    *   `cli/tools/run.rs` (or similar for other commands) is invoked.
    *   It initializes a `CliFactory` to gather dependencies like permission managers, module loaders, and the HTTP client.
3.  **Worker Creation (`runtime/worker.rs`)**: A `MainWorker` is created. This worker initializes the V8 Isolate, loads the snapshot (which includes the base JS runtime code), and sets up the op handlers.
4.  **Module Loading**: The `MainWorker` uses a `ModuleLoader` (often backed by `deno_graph` and `cli/module_loader.rs`) to resolve the entry point module (e.g., `main.ts`).
5.  **Execution**: The runtime invokes the main script within the V8 isolate. Control passes between V8 (JS) and Rust via "ops" when JS code calls Deno APIs (e.g., `Deno.readTextFile`).

## Section 4: Deep Dive into 3+ Subsystems

### 1. Runtime Ops System (The JS-to-Rust bridge)
*   **Location**: Defined primarily in `libs/core` and used throughout `ext/` and `runtime/`.
*   **Core Concepts**: `#[op]` macro (using `serde_v8` for serialization/deserialization between V8 and Rust), `OpState` (shared state container), and `Extension` (collection of ops).
*   **Connection**: All runtime APIs (FS, Fetch, etc.) are implemented as ops. Extensions register these ops with the `Isolate` during initialization.

### 2. Permission System
*   **Location**: `runtime/permissions.rs`.
*   **Core Concepts**: `PermissionsContainer` and `Permissions` structs. They manage flags like `--allow-read`, `--allow-net`.
*   **Connection**: Ops check `OpState` to verify permissions before performing sensitive operations (e.g., `ext/fs` ops check `Permissions::check_read`).

### 3. Module Resolution and Loading
*   **Location**: `cli/module_loader.rs` and `libs/resolver/`.
*   **Core Concepts**: `ModuleLoader` trait, `ModuleSpecifier`, and `deno_graph`. It handles resolution (handling `import_map`, `jsr:`, `https:`) and fetching (local/remote).
*   **Connection**: Orchestrated by the `Worker` when booting the runtime or dynamically importing modules.

## Section 5: Design Decisions & Architectural Patterns

1.  **Op-Based API Design**: By forcing all runtime APIs through a standardized op interface, Deno achieves a highly secure, modular architecture. It makes it trivial to gate APIs behind permissions.
    *   *Implemented in*: `libs/core` and all `ext/*` crates.
2.  **Snapshotting**: V8 snapshots are used to serialize the base runtime JS code. This drastically improves startup performance compared to JIT-compiling the entire runtime on boot.
    *   *Implemented in*: `cli/snapshot/` and `libs/core`.
3.  **Extension Model**: Extensions encapsulate related functionality (ops, JS resources, and setup logic) into cohesive, swappable components.
    *   *Implemented in*: `ext/*`.

## Section 6: Critical Files Index

1.  `cli/main.rs`: CLI binary entry point.
2.  `cli/tools/run.rs`: Logic for executing Deno scripts.
3.  `runtime/worker.rs`: Core `Worker` abstraction for script execution.
4.  `runtime/permissions.rs`: Logic for checking runtime security permissions.
5.  `libs/core/lib.rs`: Fundamental V8 integration and op-system implementation.
6.  `cli/module_loader.rs`: High-level module loading orchestration.
7.  `libs/serde_v8/lib.rs`: Serialization bridge between JS/V8 and Rust.
8.  `ext/node/lib.rs`: Entry point for Node.js compatibility layer.
9.  `ext/fetch/lib.rs`: Implementation of `fetch` API.
10. `ext/fs/lib.rs`: Implementation of filesystem APIs.
11. `ext/http/lib.rs`: HTTP server implementation.
12. `libs/eszip/lib.rs`: Module bundling engine.
13. `libs/npm/lib.rs`: npm package management logic.
14. `Cargo.toml` (root): Central dependency version management.
15. `cli/args/mod.rs`: CLI argument parsing definitions.
16. `ext/web/lib.rs`: Base web API implementation (e.g., `TextEncoder`, `Blob`).
17. `runtime/worker_bootstrap.rs`: JS-side bootstrapping logic for workers.
18. `libs/dcore/lib.rs`: Deno-specific core utilities.
