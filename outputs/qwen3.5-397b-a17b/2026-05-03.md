### Section 5: Design Decisions & Architectural Patterns

#### 5.1 Extension-Based Architecture

**Decision:** Deno uses a modular extension system (`deno_core::extension!`) to organize functionality.

**Rationale:**
- **Separation of Concerns:** Each extension (web, http, node, etc.) is self-contained
- **Lazy Loading:** Extensions can be loaded on-demand
- **Testability:** Extensions can be tested independently
- **Clear Dependencies:** Extension dependencies are explicit in macro invocation

**Implementation:**
```rust
deno_core::extension!(deno_http,
  deps = [deno_web, deno_net, deno_fetch, deno_websocket],
  ops = [op_http_accept, op_http_serve, ...],
  esm = ["00_http.js"],
);
```

**Location:** All `ext/*/lib.rs` files, `libs/core/extensions.rs`

---

#### 5.2 Permission-First Security Model

**Decision:** All potentially dangerous operations require explicit permission checks.

**Rationale:**
- **Security by Default:** No access without explicit grant
- **Granular Control:** Fine-grained permissions (path-level, host-level)
- **Audit Trail:** Permission checks are logged and traceable

**Implementation:**
- `PermissionsContainer` wraps all permission state
- Every dangerous op calls `check_*()` before execution
- Permissions can be: granted, denied, or prompt

**Location:** `runtime/permissions.rs`, all `ext/*/ops.rs` files

---

#### 5.3 Snapshot-Based Startup Optimization

**Decision:** Deno uses V8 snapshots to reduce startup time.

**Rationale:**
- **Fast Cold Start:** Pre-compiled JS avoids re-parsing
- **Reduced Memory:** Shared read-only data
- **Consistent State:** All users start from same baseline

**Implementation:**
- `cli/snapshot/` - Snapshot generation
- `runtime/snapshot.rs` - Snapshot loading
- Feature flags: `include_js_files_for_snapshotting`, `hmr`

**Trade-off:** Snapshot updates require rebuild; HMR mode disables snapshots for development.

---

#### 5.4 Node.js Compatibility Layer

**Decision:** Extensive Node.js compatibility via `deno_node` extension.

**Rationale:**
- **Ecosystem Access:** Leverage npm packages
- **Migration Path:** Easier transition from Node.js
- **Standards Compliance:** Node APIs are well-documented

**Implementation:**
- `ext/node/` - Node.js API implementations
- `libs/node_resolver/` - Node module resolution
- `libs/npm/` - npm package resolution
- `node_modules/` virtual file system

**Key Pattern:** Polyfill Node.js built-ins using Deno's internal APIs.

---

#### 5.5 Op-Based JS ↔ Rust Bridge

**Decision:** JavaScript calls Rust functions via "ops" (operations).

**Rationale:**
- **Performance:** Direct V8 integration, minimal overhead
- **Type Safety:** `op2` macro generates type-safe bindings
- **Ergonomics:** Simple annotation-based API definition

**Implementation:**
```rust
#[op2(fast)]
fn op_add(a: i64, b: i64) -> i64 {
  a + b
}
```

**Location:** `libs/ops/`, all `ext/*/ops.rs` files
