### 4.1 Runtime Ops System (`libs/ops/`, `libs/core/`)

The ops system is Deno's primary mechanism for JavaScript-to-Rust interop.

**Key Components:**

1. **`#[op2]` Macro** (`libs/ops/op2/mod.rs`):
   - Replaces the older `#[op]` macro
   - Generates optimized V8 function bindings
   - Supports fastcalls (fast path for simple types)
   - Supports async operations (eager, lazy, deferred)
   - Generates CppGC objects for native JS classes

2. **Op Definition Pattern** (`runtime/ops/bootstrap.rs`):
```rust

    std::process::id()
}
```

3. **Extension Registration** (`deno_core::extension!` macro):

## Section 5: Design Decisions & Architectural Patterns

### 5.1 Extension-Based Architecture

**Decision**: Runtime APIs are implemented as independent "extensions" that are registered with `deno_core::extension!` macro.

**Rationale**:
- **Modularity**: Each API is self-contained
- **Composability**: Embedders can choose which extensions to include
- **Testability**: Extensions can be tested in isolation
- **Snapshotting**: Extensions can be pre-compiled into V8 snapshots

**Implementation**: `libs/core/extensions.rs` defines the `Extension` struct and `extension!` macro.

### 5.2 Op2 Macro for Type-Safe Bindings

**Decision**: Use proc-macro (`#[op2]`) instead of manual op registration for JS↔Rust interop.

**Rationale**:
- **Performance**: Generates optimized fastcall paths automatically
- **Type Safety**: Compile-time checking of argument/return types
- **Ergonomics**: Reduces boilerplate significantly
- **CppGC Support**: Native class support with garbage collection

**Implementation**: `libs/ops/op2/mod.rs` and submodules generate V8 bindings at compile time.

### 5.3 Lazy Factory Pattern for Dependency Initialization

**Decision**: `CliFactory` in `cli/factory.rs` uses `Deferred<T>` pattern for lazy initialization.

**Rationale**:
- **Performance**: Expensive resources (HTTP clients, caches) only created when needed
- **Resource efficiency**: Some CLI commands don't need all services
- **Order independence**: Services can be initialized in any order

**Implementation**: `Deferred::get_or_try_init_async()` caches results in `OnceCell`.

### 5.4 Snapshot-Based Startup Optimization

**Decision**: Pre-compile JavaScript bootstrap code into V8 snapshots.

**Rationale**:
- **Fast startup**: Avoids parsing/compiling common code on every run
- **Reduced I/O**: Bootstrap JS embedded in binary
- **Security**: Smaller attack surface

**Implementation**: Build scripts in `cli/snapshot/` generate snapshots; runtime feature flags enable/disable.

### 5.5 Separation of deno_core and deno_runtime

**Decision**: Split between low-level `deno_core` (V8 primitives) and `deno_runtime` (full runtime).

**Rationale**:
- **Embedder Support**: Others can build custom runtimes with `deno_core`
- **Testing**: Core can be tested without full runtime
- **Flexibility**: Different deployment scenarios can customize

**Implementation**: `libs/core/` contains pure V8 bindings; `runtime/` adds extensions and worker logic.

### 5.6 System Trait Abstraction (`sys_traits`)

**Decision**: Abstract system operations behind traits for testing and cross-platform support.

**Rationale**:
- **Testability**: Mock file system, env vars, time, etc.
- **Cross-platform**: Different implementations for Windows/Unix
- **Security**: Sandboxed environments can provide restricted implementations

**Implementation**: `sys_traits` crate defines `Fs`, `Env`, `Http`, `Time`, etc. traits.
