### Section 5: Design Decisions & Architectural Patterns

#### **1. Extension-Based Architecture**
**Decision:** Modular system with 30+ extensions for different APIs
**Rationale:** Isolation, tree-shaking, testability
**Implementation:** `deno_core::extension!` macro in each `ext/*/lib.rs`

#### **2. Op-Based Bridge Pattern**
**Decision:** All JS-to-Rust calls go through "ops"
**Rationale:** Security, performance, unified permission checking
**Implementation:** `deno_core::op2` macro generates ops from Rust functions

#### **3. Permission-First Security**
**Decision:** Permissions checked at op boundary, not capability-based
**Rationale:** Easy to audit, user-friendly prompts
**Implementation:** `runtime/permissions/lib.rs:PermissionsContainer`

#### **4. Module Graph Compilation**
**Decision:** Build complete dependency graph before execution
**Rationale:** Parallel fetching, type-checking, error aggregation
**Implementation:** `deno_graph` crate and `cli/graph_container.rs`

#### **5. Snapshot-Based Startup**
**Decision:** Pre-compile JavaScript bindings into V8 snapshots
**Rationale:** 10-100x faster startup
**Implementation:** `cli/snapshot/` and `runtime/snapshot.rs`
