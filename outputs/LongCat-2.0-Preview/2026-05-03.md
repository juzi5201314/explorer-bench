- `CliNpmResolver` (`cli/npm.rs`) â€” Wraps `NpmResolver`, provides `resolve_pkg_folder_from_deno_module_req()`.
- `CliNpmInstaller` (`cli/npm.rs`) â€” Wraps `NpmInstaller` from `libs/npm_installer/`.
- `NodeRequireLoader` trait (`ext/node/lib.rs:53`) â€” Interface for loading CommonJS modules. Implemented by `CliNodeRequireLoader`.

**Flow:**
1. When an `npm:` specifier is encountered, `CliNpmResolver` resolves it to a folder in `node_modules/`.
2. `maybe_npm_install()` in `cli/tools/run/mod.rs` runs `npm install` if needed.
3. `NodeResolver` resolves `node:` builtins and package-internal imports.
4. CJS modules are handled by the `require()` polyfill in `ext/node/polyfills/`, which uses ops like `op_require_resolve_exports`, `op_require_read_file`, `op_vm_create_script`.

## Section 5: Design Decisions & Architectural Patterns

### 1. Extension-Based Architecture (V8 Isolate Modularity)

**Decision:** Every API surface (fs, net, http, crypto, node compat, etc.) is a separate `Extension` crate that registers Rust ops and JavaScript modules via `deno_core::extension!()`.

**Rationale:** This enables:
- **Snapshotting:** All JS source code is compiled into a V8 startup snapshot, eliminating runtime JS parsing overhead.
- **Selective inclusion:** `denort` (the lighter runtime) can include only the extensions it needs.
- **Compilation boundaries:** Each extension compiles independently, improving incremental build times.
- **Testability:** Extensions can be tested in isolation via `deno_core_testing`.

**Implementation:** `runtime/worker.rs:common_extensions()` assembles all extensions. The `extension!()` macro generates the registration code that calls `Extension::register_ops_and_esm()`.

### 2. op2 Proc Macro for Zero-Cost V8â†”Rust Dispatch

**Decision:** All Rust ops use the `#[op2]` proc macro (in `libs/ops/op2/`) rather than hand-written V8 bindings or the older `deno_core::op!` macro.

**Rationale:**
- **Type safety:** The macro generates type-safe argument deserialization from V8 values at compile time.
- **Fast path:** Automatically generates V8 Fast API call variants for ops with primitive parameter types, bypassing the expensive V8 API.
- **Consistency:** All ops follow the same pattern, making the codebase maintainable.
- **Metadata:** Ops carry metadata (name, async, reentrant, no-side-effect) used by the runtime for optimization and tooling.

**Implementation:** `op2!` generates:
- A slow dispatch function (full V8 API, handles all types including ser/de types via `serde_v8`)
- A fast dispatch function (V8 Fast API, limited to primitive types)
- An async dispatch wrapper (spawns on Tokio runtime, returns promises)
- The `OpDecl` constant with function pointers and metadata

### 3. Lazy/Deferred Initialization via `CliFactory`

**Decision:** The CLI uses a `CliFactory` pattern (`cli/factory.rs`) where all services are wrapped in `Deferred<T>` (a `OnceCell` variant) and initialized lazily on first access.

**Rationale:**
- **Subcommand flexibility:** Different subcommands need different services. `deno lint` doesn't need npm resolution; `deno test` doesn't need compilation.
- **Startup performance:** Only the services actually needed are initialized.
- **Dependency ordering:** `Deferred` handles complex initialization order without explicit sequencing.

**Implementation:** `CliFactory` has methods like `npm_resolver()`, `file_fetcher()`, `type_checker()` each returning `Result<&T, AnyError>`. The first call initializes the service; subsequent calls return the cached reference. `get_or_try_init_async()` supports async initialization.

### 4. Dual TypeScript Compilation (tsgo + JS tsc)

**Decision:** Deno v2.7 uses **tsgo** (a Go-based TypeScript compiler) as the primary type checker, with the legacy JS-based TypeScript (`00_typescript.js`) as a fallback.

**Rationale:**
- **Performance:** tsgo provides ~10x faster type checking compared to the JS-based tsc.
- **Compatibility:** tsgo implements the same TypeScript API surface, communicating via `--api` mode with callbacks for file reading, module resolution, etc.
- **Migration path:** The `ensure_tsgo()` function downloads the tsgo binary on first use. If unavailable, it falls back to JS tsc.

**Implementation:** `cli/tsc/go.rs` spawns the tsgo binary as a child process with `SyncRpcChannel`. The `Handler` struct implements `CallbackHandler` to resolve modules, read files, and provide package.json scopes back to tsgo. A synthetic `tsconfig.json` is created in memory with the module graph's root files.

### 5. Permission Quadri-State with Partial Grants

**Decision:** Permissions use a 6-state enum (`Granted`, `GrantedPartial`, `Prompt`, `Denied`, `DeniedPartial`, `Ignored`) rather than a simple boolean.

**Rationale:**
- **Granular control:** Users can `--allow-read` globally but `--deny-read=/etc/shadow` for specific paths. This creates `GrantedPartial` state.
- **Composability:** The `--allow-*` and `--deny-*` flags interact predictably. Deny takes precedence.
- **Auditability:** The `AUDIT_SINK` can log every permission check, including partial grants/denials, for security auditing.
- **Broker support:** The `PermissionBroker` (`runtime/permissions/broker.rs`) allows delegating permission decisions to a parent process via IPC.

**Implementation:** `runtime/permissions/lib.rs` defines `PermissionState` and the `AllowPartial` enum. Each permission type (Read, Write, Net, etc.) has its own check logic that handles partial matching (e.g., URL prefix matching for net permissions).
